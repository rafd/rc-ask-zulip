## Authentication

The Recurse Center API provides three authentication methods that serve different classes of applications.

* OAuth 2.0 - web apps
* Personal access tokens - mobile and native apps, and ad-hoc experimentation
* Basic auth - used to create personal access tokens

Nearly every API resource requires authentication of some type. If you're making an app that uses OAuth, you can [register your app on recurse.com](https://www.recurse.com/settings/apps).

### OAuth 2.0

The Recurse Center API allows [OAuth 2.0](http://oauth.net/2/) for authentication. When you write an app that uses OAuth, `www.recurse.com` asks a user for permission for your app to use the API on their behalf. If the user gives permission, `www.recurse.com` will give your app an access token that allows you to use the API as that user. The user can revoke your app's permission at any time. Once you have an access token, you can use it either in the Authorization HTTP header or in the `access_token` query parameter to make API requests. For details on how to do this, see the "Making requests" section further down in this page.

Describing how OAuth works in detail is beyond the scope of this documentation. Most programming langauges have OAuth 2 clients that automate most of the complicated parts of this process in a few lines of code.

The authorization endpoint is `/oauth/authorize`. Authorization codes are the only supported authorization grant type. The token endpoint is `/oauth/token`. Access tokens are good for two hours. The token endpoint also gives refresh tokens that can be used to request new access tokens. Refresh tokens can only be used once: when you use a refresh token, you will be given a new access token and a new refresh token. Your OAuth 2 client library should be able to handle this automatically. You can make a new application [on your settings page](https://www.recurse.com/settings/apps).

The redirect URI submitted with the authorization request must exactly match the redirect URI associated with your app on www.recurse.com. If you are experimenting with an OAuth client at the command line and don't yet have an HTTP server that the user can be redirected to, you can use `urn:ietf:wg:oauth:2.0:oob` as your redirect URI, which will cause www.recurse.com to render the authorization code in the browser.

OAuth should not be used for native apps and mobile apps since it is not possible to keep your client secret hidden from users who have your code, compiled or otherwise.

### Personal access tokens

A personal access token is a randomly-generated token that can be used to make API requests. They are used in the same way as OAuth access token (either in the Authorization header using the Bearer authentication type or in the `access_token` query parameter), but unlike OAuth access tokens, they don't expire. Personal access tokens are useful for native applications where it's not possible to securely store an OAuth client secret, and for exploring the API by hand.

There are two ways to create a personal access token: you can make one manually in the [Apps section of your settings page](https://www.recurse.com/settings/apps) or you can create them programatically using the tokens endpoint described below. The tokens endpoint is accessed using the Basic authentication type, allowing you to exchange a username and password for a personal access token that can be used with the rest of the API. Personal access tokens can be revoked at any time from the Apps section of your settings page.

### Native and mobile applications

Native and mobile applications should not use OAuth 2.0. This is because it is impossible to keep the client secret hidden when your users have access to your code on their device. Instead, native and mobile apps should use the tokens endpoint to exchange the user's email address and password for a personal access token. When you create the personal access token, you should set its description to be the name of your app. You should never store the user's password on the device.

## Versions

The Recurse Center API is versioned. The current version is v1. Once an API version is released, there will be no backwards incompatible changes made to that version. ### Profiles

#### Search

    GET /api/v1/profiles

This is the endpoint used by the [RC Directory](https://www.recurse.com/directory). It is used to search and filter members of the RC community.

##### Query parameters

* `query` (String) – Search by name, skills, profile questions. An empty query will cause results to be returned in descending order by profile creation date.
* `batch_id` (Integer) – Filter to people who have a stint for this batch ID.
* `location_id` (Integer) – Filter to people who live near this location (either close by if it's a city, or within a country or continent).
* `role` (String) – Filter to people who have a stint of a certain type.
    * `recurser`
    * `resident`
    * `facilitator`
    * `faculty`
* `scope` (String) – Narrow the search space to a certain group of people.
    * `current` — People currently at RC.
    * `overlap` – People who overlapped with the current user at RC.
* `limit` (Integer) – Limit the number of results returned. Default 20, max 50.
* `offset` (Integer) – Offset the results by provided value for purposes of pagination.

### Personal access tokens

#### Create a personal access token

    POST /api/v1/tokens

Create a personal access token.

##### Authentication

The tokens resource is special because it uses the [HTTP Basic authentication](https://tools.ietf.org/html/rfc7617) method. This allows native and mobile apps to exchange a username and password for a personal access token. Here's an example of an Authorization header using basic auth:

    # Base64 encoded nick@recurse.com:password
    Authorization: Basic bmlja0ByZWN1cnNlLmNvbTpwYXNzd29yZA==

You should not save a user's password after creating a token.

##### Request parameters

* `description` (String) - A user-visible description for this personal access token. Should be the name of your app. (Required).

### Hub visits

A hub visit indicates that a person is (or plans to be) at the hub on a given date. Hub visits have optional notes (e.g., "I'll be in from 9am to around 3pm" or "I'm on the 4th floor working on my compiler in Rust and would love to pair"). They also have optional `app_data`, which is a JSON blob of arbitrary application-specific metadata. You can use this to store small amounts of data and communicate across apps. Hub visits have timestamps to indicate when they were created and last updated, as well as fields that indicate which app (if any) created and/or last updated the visit.

All RC community members can view all hub visits. You can only create, update, and delete your own hub visits. You can give an app permission to create, update, and delete your hub visits via OAuth or a personal access token.

#### Get hub visits

- `date` (Date) - only show visits on this date
